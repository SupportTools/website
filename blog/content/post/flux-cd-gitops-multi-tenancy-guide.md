Platform teams managing multiple application teams face a consistent challenge: how to provide GitOps self-service without granting tenants the ability to affect each other's workloads or the platform itself. Flux CD's multi-tenancy model provides the building blocks to solve this — namespace-scoped sources, impersonation via ServiceAccount, hierarchical Kustomizations, and a notification engine that pushes delivery status back to source control. This guide covers every layer from cluster bootstrap through image automation and cross-cluster push mode.

# Flux CD Multi-Tenancy: GitOps for Platform Teams and Tenant Isolation

## Architecture Overview

A properly isolated Flux multi-tenancy setup separates concerns across three layers:

- **Platform layer** — managed by the platform team, controls Flux itself, cluster-wide policies, and shared infrastructure
- **Tenant namespace layer** — each tenant team owns one or more namespaces; their Flux resources are scoped to those namespaces
- **Source layer** — each tenant's GitRepository or OCIRepository is created in the tenant namespace, not the flux-system namespace

The key enforcement mechanism is the `spec.serviceAccountName` field on a Kustomization or HelmRelease. When set, Flux impersonates that ServiceAccount for all Kubernetes API calls generated by that reconciliation. By ensuring the ServiceAccount has only namespace-scoped permissions, the platform team enforces isolation without relying solely on admission control.

```
cluster-admin (platform team)
  └── flux-system namespace
        ├── GitRepository: platform-config (cluster-wide)
        ├── Kustomization: platform-infra (no SA impersonation)
        └── Kustomization: tenant-onboarding (creates tenant namespaces + SA)
              └── tenant-a namespace
                    ├── ServiceAccount: flux-reconciler (namespace RBAC only)
                    ├── GitRepository: tenant-a-app
                    └── Kustomization: tenant-a-workloads (impersonates flux-reconciler)
```

## Bootstrap and Flux Installation

### Flux CLI Bootstrap with GitHub

```bash
# Install Flux CLI
curl -s https://fluxcd.io/install.sh | sudo bash

# Bootstrap Flux onto the cluster using a GitHub repository
# The --components-extra flag adds the image automation controllers
flux bootstrap github \
  --owner=my-org \
  --repository=fleet-infra \
  --branch=main \
  --path=clusters/production \
  --components-extra=image-reflector-controller,image-automation-controller \
  --personal=false \
  --token-auth=false

# Verify all Flux controllers are running
flux check

# Expected output:
# ► checking prerequisites
# ✔ Kubernetes 1.29.x >=1.26.0-0
# ► checking controllers
# ✔ helm-controller: deployment ready
# ✔ kustomize-controller: deployment ready
# ✔ notification-controller: deployment ready
# ✔ source-controller: deployment ready
# ✔ image-reflector-controller: deployment ready
# ✔ image-automation-controller: deployment ready
```

### SOPS Encryption for Secrets in Git

```bash
# Create a GPG key for SOPS (or use age)
age-keygen -o age.agekey

# Store the private key as a cluster secret
cat age.agekey |
kubectl create secret generic sops-age \
  --namespace=flux-system \
  --from-file=age.agekey=/dev/stdin

# Configure Flux kustomize-controller to use SOPS
# clusters/production/flux-system/kustomization.yaml
```

```yaml
# clusters/production/flux-system/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - gotk-components.yaml
  - gotk-sync.yaml
patches:
  - patch: |
      - op: add
        path: /spec/template/spec/containers/0/env/-
        value:
          name: SOPS_AGE_KEY_FILE
          value: /etc/fluxcd/age/age.agekey
      - op: add
        path: /spec/template/spec/volumes/-
        value:
          name: sops-age
          secret:
            secretName: sops-age
      - op: add
        path: /spec/template/spec/containers/0/volumeMounts/-
        value:
          name: sops-age
          mountPath: /etc/fluxcd/age
          readOnly: true
    target:
      kind: Deployment
      name: kustomize-controller
```

## Platform-Level Source Configuration

### Cluster-Wide GitRepository for Platform Configuration

```yaml
# clusters/production/platform/gitrepository-platform.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: platform-config
  namespace: flux-system
spec:
  interval: 1m
  # Reference to the main platform config repository
  url: https://github.com/my-org/platform-config
  ref:
    branch: main
  # Secret must be in flux-system namespace for cluster-scoped source
  secretRef:
    name: github-platform-token
  # Ignore generated files to prevent reconciliation loops
  ignore: |
    # exclude all
    /*
    # include base and clusters directories
    !/base/
    !/clusters/
```

### OCIRepository Source for Helm Charts

OCIRepository allows pulling Helm charts and manifests directly from an OCI-compatible registry, removing the dependency on a Git server for chart distribution.

```yaml
# clusters/production/platform/ocirepository-charts.yaml
apiVersion: source.toolkit.fluxcd.io/v1beta2
kind: OCIRepository
metadata:
  name: platform-charts
  namespace: flux-system
spec:
  interval: 5m
  # OCI registry URL following the oci:// scheme
  url: oci://ghcr.io/my-org/charts
  ref:
    # Pin to a semver tag rather than mutable latest
    semver: ">=1.0.0 <2.0.0"
  # Registry credentials stored in flux-system
  secretRef:
    name: ghcr-credentials
  # Verify the image signature using cosign
  verify:
    provider: cosign
    secretRef:
      name: cosign-public-key
```

### Platform Kustomization (No Impersonation)

The platform team's own Kustomizations run with the kustomize-controller's full cluster-admin permissions. No `serviceAccountName` is set, which means the controller uses its own ServiceAccount.

```yaml
# clusters/production/platform/kustomization-platform-infra.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: platform-infra
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: platform-config
  path: ./infrastructure
  prune: true
  # Wait for all resources to be healthy before marking reconciliation complete
  wait: true
  timeout: 5m
  # Decrypt SOPS-encrypted secrets
  decryption:
    provider: sops
    secretRef:
      name: sops-age
  # Health checks on critical infrastructure CRDs
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: cert-manager
      namespace: cert-manager
    - apiVersion: apps/v1
      kind: Deployment
      name: ingress-nginx
      namespace: ingress-nginx
```

## Tenant Onboarding

### Tenant Namespace and RBAC Manifest

The platform team maintains a directory of tenant onboarding manifests. Adding a new tenant is a pull request to this directory.

```yaml
# infrastructure/tenants/tenant-a/namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: tenant-a
  labels:
    # Label used by network policies to identify tenant namespaces
    tenant: "true"
    tenant-id: "tenant-a"
    # Pod Security Standards enforcement at namespace level
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest
---
# ServiceAccount that the tenant's Flux Kustomizations will impersonate
apiVersion: v1
kind: ServiceAccount
metadata:
  name: flux-reconciler
  namespace: tenant-a
---
# Role scoped to tenant-a namespace only
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: flux-reconciler
  namespace: tenant-a
rules:
  # Standard workload resources
  - apiGroups: [""]
    resources: ["namespaces", "serviceaccounts", "services", "configmaps", "secrets",
                 "persistentvolumeclaims", "pods"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["apps"]
    resources: ["deployments", "statefulsets", "daemonsets", "replicasets"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["batch"]
    resources: ["jobs", "cronjobs"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["networking.k8s.io"]
    resources: ["ingresses", "networkpolicies"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["policy"]
    resources: ["poddisruptionbudgets"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Flux HelmRelease requires access to its own CRDs in the namespace
  - apiGroups: ["helm.toolkit.fluxcd.io"]
    resources: ["helmreleases"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Flux Kustomization status updates
  - apiGroups: ["kustomize.toolkit.fluxcd.io"]
    resources: ["kustomizations"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: flux-reconciler
  namespace: tenant-a
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: flux-reconciler
subjects:
  - kind: ServiceAccount
    name: flux-reconciler
    namespace: tenant-a
```

### Network Policy for Tenant Isolation

```yaml
# infrastructure/tenants/tenant-a/networkpolicy.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: tenant-isolation
  namespace: tenant-a
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
  ingress:
    # Allow traffic only from within the same tenant namespace
    - from:
        - podSelector: {}
    # Allow ingress from nginx ingress controller
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
  egress:
    # Allow DNS resolution
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
      ports:
        - protocol: UDP
          port: 53
    # Allow egress within namespace
    - to:
        - podSelector: {}
    # Allow egress to external services (can be restricted further)
    - to:
        - ipBlock:
            cidr: 0.0.0.0/0
            except:
              # Block access to other tenant namespaces via ClusterIP range
              - 10.96.0.0/12
```

## Tenant-Level Flux Sources

### GitRepository in Tenant Namespace

Tenant sources live in the tenant namespace, not flux-system. This is enforced by the platform team's RBAC — tenants cannot create Sources in flux-system.

```yaml
# infrastructure/tenants/tenant-a/gitrepository-tenant-a.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: tenant-a-app
  namespace: tenant-a
spec:
  interval: 1m
  url: https://github.com/my-org/tenant-a-app
  ref:
    branch: main
  # Secret in tenant-a namespace — tenant manages their own credentials
  secretRef:
    name: github-tenant-a-token
```

### Tenant Kustomization with ServiceAccount Impersonation

```yaml
# infrastructure/tenants/tenant-a/kustomization-tenant-a-workloads.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: tenant-a-workloads
  namespace: flux-system   # Kustomization lives in flux-system
spec:
  interval: 5m
  sourceRef:
    kind: GitRepository
    name: tenant-a-app
    # Cross-namespace reference: source is in tenant-a
    namespace: tenant-a
  path: ./deploy/production
  prune: true
  # CRITICAL: impersonate the tenant's ServiceAccount
  # All Kubernetes API calls will be made as flux-reconciler in tenant-a
  # This limits what Flux can create/modify to what that SA is authorized for
  serviceAccountName: flux-reconciler
  # Target namespace enforces all resources go to tenant-a
  targetNamespace: tenant-a
  # Force namespace on all resources even if they specify a different one
  force: false
  # Patch to inject the environment label on all deployments
  patches:
    - patch: |
        - op: add
          path: /metadata/labels/environment
          value: production
      target:
        kind: Deployment
  # Decrypt secrets if tenant uses SOPS
  decryption:
    provider: sops
    secretRef:
      name: sops-age
      namespace: flux-system
  # Dependency: platform infra must be ready before tenants reconcile
  dependsOn:
    - name: platform-infra
      namespace: flux-system
```

## HelmRelease with Tenant Impersonation

Tenants can also deploy Helm charts. The HelmRelease references a HelmRepository or OCIRepository source within the tenant namespace.

```yaml
# tenant-a-app/deploy/production/helmrelease-api.yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: tenant-a-api
  namespace: tenant-a
spec:
  interval: 10m
  chart:
    spec:
      chart: api-service
      version: ">=1.2.0 <2.0.0"
      sourceRef:
        kind: HelmRepository
        name: tenant-a-charts
        namespace: tenant-a
  # Maximum time for a Helm operation to complete
  timeout: 5m
  # Install configuration
  install:
    remediation:
      # Retry failed installs up to 3 times
      retries: 3
    createNamespace: false  # Namespace already exists
  # Upgrade configuration
  upgrade:
    cleanupOnFail: true
    remediation:
      retries: 3
      remediateLastFailure: true
  # Rollback if upgrade fails
  rollback:
    timeout: 5m
    cleanupOnFail: true
  values:
    replicaCount: 2
    image:
      repository: ghcr.io/my-org/tenant-a-api
      # Marker updated by image automation
      tag: "1.5.3" # {"$imagepolicy": "tenant-a:tenant-a-api"}
    resources:
      requests:
        cpu: 100m
        memory: 128Mi
      limits:
        memory: 256Mi
    serviceAccount:
      # Use the existing SA, do not create a new one
      create: false
      name: tenant-a-api
  # Override specific values from a ConfigMap or Secret
  valuesFrom:
    - kind: ConfigMap
      name: tenant-a-api-config
      valuesKey: values.yaml
    - kind: Secret
      name: tenant-a-api-secrets
      valuesKey: secrets.yaml
      optional: true
```

## Image Automation

Image automation allows Flux to detect new container image tags in a registry and automatically update the Git repository with the new tag. This enables a full GitOps loop without manual version bumping.

### ImageRepository

```yaml
# infrastructure/tenants/tenant-a/image-automation/imagerepository-tenant-a-api.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageRepository
metadata:
  name: tenant-a-api
  namespace: tenant-a
spec:
  # Container registry to scan
  image: ghcr.io/my-org/tenant-a-api
  # How often to poll for new tags
  interval: 1m
  # Secret with registry credentials for scanning private registries
  secretRef:
    name: ghcr-credentials
  # Exclude tags matching this regex (e.g., dev builds, PRs)
  exclusionList:
    - "^.*\\.dev$"
    - "^pr-[0-9]+$"
    - "^main-[a-f0-9]{7}$"
```

### ImagePolicy

ImagePolicy defines which tags to consider valid and selects the latest matching one.

```yaml
# infrastructure/tenants/tenant-a/image-automation/imagepolicy-tenant-a-api.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: tenant-a-api
  namespace: tenant-a
spec:
  imageRepositoryRef:
    name: tenant-a-api
  # Select the latest tag matching semver constraints
  policy:
    semver:
      range: ">=1.0.0 <2.0.0"
  # Alternative: filter by tag prefix before applying semver
  filterTags:
    pattern: "^(?P<ts>[0-9]+)-(?P<ver>[0-9]+\\.[0-9]+\\.[0-9]+)$"
    extract: "$ver"
```

### ImageUpdateAutomation

```yaml
# infrastructure/tenants/tenant-a/image-automation/imageupdateautomation.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageUpdateAutomation
metadata:
  name: tenant-a-flux-image-updates
  namespace: tenant-a
spec:
  interval: 1m
  # Source repository to update
  sourceRef:
    kind: GitRepository
    name: tenant-a-app
  # Git branch to commit image updates to
  git:
    checkout:
      ref:
        branch: main
    commit:
      author:
        email: flux-bot@myorg.com
        name: Flux Bot
      # Commit message template
      messageTemplate: |
        chore(image): update {{range .Updated.Images}}{{.}}{{end}}

        Updated image policies:
        {{range .Updated.Images -}}
        - {{.}} ({{.Identifier}})
        {{end -}}
    push:
      branch: main
  # Which files to scan for image policy markers
  update:
    strategy: Setters
    path: ./deploy/production
```

The marker in the HelmRelease values `# {"$imagepolicy": "tenant-a:tenant-a-api"}` tells the image automation controller which ImagePolicy to use when updating that specific value.

## Flux Notifications

Notifications allow Flux to push reconciliation events to external systems — Slack, Microsoft Teams, GitHub commit statuses, and more.

### GitHub Commit Status Provider

```yaml
# clusters/production/platform/notification-provider-github.yaml
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Provider
metadata:
  name: github-status
  namespace: flux-system
spec:
  type: github
  address: https://github.com/my-org/fleet-infra
  # Secret containing the GitHub token with repo:status scope
  secretRef:
    name: github-token
---
# Alert that fires on all Kustomization events for production cluster
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Alert
metadata:
  name: production-status
  namespace: flux-system
spec:
  providerRef:
    name: github-status
  # Only send alerts for Warning and Error events
  eventSeverity: info
  eventSources:
    - kind: Kustomization
      namespace: flux-system
      name: "*"
  # Include details in the commit status description
  inclusionList:
    - ".*"
```

### Slack Notification for Tenant Failures

```yaml
# infrastructure/tenants/tenant-a/notification-provider-slack.yaml
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Provider
metadata:
  name: slack-tenant-a
  namespace: tenant-a
spec:
  type: slack
  channel: tenant-a-deployments
  # Secret with the Slack webhook URL

    name: slack-webhook-tenant-a
---
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Alert

  name: tenant-a-deployment-alerts
  namespace: tenant-a

  providerRef:
    name: slack-tenant-a

  eventSources:
    - kind: Kustomization
      name: "*"
    - kind: HelmRelease
      name: "*"
    - kind: ImageUpdateAutomation
      name: "*"

  suspend: false
```

## Multi-Cluster Push Mode

In multi-cluster deployments, a management cluster runs Flux and pushes workloads to remote clusters. This avoids needing Flux installed on each remote cluster.

### Kubeconfig Secret for Remote Cluster

```bash
# Extract the remote cluster kubeconfig
kubectl config view \
  --context=production-eu-west-1 \
  --minify=true \
  --flatten=true > /tmp/eu-west-1.kubeconfig

# Store it as a secret in flux-system on the management cluster
kubectl create secret generic eu-west-1-kubeconfig \
  --namespace=flux-system \
  --from-file=value=/tmp/eu-west-1.kubeconfig
```

### Kustomization Targeting Remote Cluster

```yaml
# clusters/management/remote-clusters/kustomization-eu-west-1.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: eu-west-1-workloads
  namespace: flux-system
spec:
  interval: 5m
  sourceRef:
    kind: GitRepository
    name: platform-config
  path: ./clusters/eu-west-1
  prune: true
  # Reference to the kubeconfig secret for the remote cluster
  kubeConfig:
    secretRef:
      name: eu-west-1-kubeconfig
      key: value
  # The service account is interpreted in the context of the remote cluster
  # Ensure a flux-reconciler SA exists on the remote cluster
  serviceAccountName: flux-reconciler
```


### Key Flux Metrics to Alert On

```yaml
# alerting/rules/flux-alerts.yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: flux-alerts
  namespace: flux-system
spec:
  groups:
    - name: flux.rules
      rules:
        # Alert when a Kustomization has been failing for more than 10 minutes
        - alert: FluxKustomizationNotReady
          expr: |
            gotk_reconcile_condition{type="Ready", status="False", kind="Kustomization"} == 1
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "Flux Kustomization {{ $labels.name }} is not ready"
            description: "Kustomization {{ $labels.namespace }}/{{ $labels.name }} has been failing for more than 10 minutes."

        # Alert when a HelmRelease is stuck
        - alert: FluxHelmReleaseNotReady
          expr: |
            gotk_reconcile_condition{type="Ready", status="False", kind="HelmRelease"} == 1
          for: 15m
          labels:
            severity: warning
          annotations:
            summary: "Flux HelmRelease {{ $labels.name }} is not ready"

        # Alert when reconcile duration is elevated (possible API server pressure)
        - alert: FluxReconcileDurationHigh
          expr: |
            histogram_quantile(0.99,
              rate(gotk_reconcile_duration_seconds_bucket[5m])
            ) > 60
          for: 5m
          labels:
            severity: info
          annotations:
            summary: "Flux reconcile duration is high for {{ $labels.kind }}"
```

## Operational Runbook

### Suspending and Resuming Reconciliation

```bash
# Suspend all reconciliation for a tenant during an incident
flux suspend kustomization tenant-a-workloads --namespace flux-system

# Suspend a specific HelmRelease to prevent automated rollbacks
flux suspend helmrelease tenant-a-api --namespace tenant-a

# Resume after incident resolution
flux resume kustomization tenant-a-workloads --namespace flux-system
flux resume helmrelease tenant-a-api --namespace tenant-a
```

### Forcing Immediate Reconciliation

```bash
# Force reconciliation without waiting for the next interval
flux reconcile kustomization tenant-a-workloads --namespace flux-system --with-source

# Force reconciliation of a HelmRelease
flux reconcile helmrelease tenant-a-api --namespace tenant-a

# Force source refresh (useful after pushing to Git)
flux reconcile source git tenant-a-app --namespace tenant-a
```

### Debugging Failed Reconciliation

```bash
# Get detailed status of all Flux resources
flux get all --all-namespaces

# Get specific failing Kustomization details
kubectl describe kustomization tenant-a-workloads --namespace flux-system

# Stream Flux controller logs for debugging
kubectl logs -n flux-system deployment/kustomize-controller -f | \
  jq 'select(.msg | contains("tenant-a"))'

# Check the generated manifests before they are applied
flux diff kustomization tenant-a-workloads --namespace flux-system

# Export the last applied state from the Git source
flux export kustomization tenant-a-workloads --namespace flux-system

# Verify SOPS decryption is working
flux check --pre
kubectl -n flux-system get secret sops-age -o jsonpath='{.data.age\.agekey}' | base64 -d
```

### Image Automation Status

```bash
# Check current image policy status
flux get image policy tenant-a-api --namespace tenant-a

# Check ImageRepository scan results
kubectl get imagerepository tenant-a-api --namespace tenant-a -o jsonpath='{.status}'

# Force an immediate image scan
flux reconcile image repository tenant-a-api --namespace tenant-a

# Check what the image automation last committed
kubectl describe imageupdateautomation tenant-a-flux-image-updates --namespace tenant-a
```

## Tenant Onboarding Automation

A GitOps-native onboarding process allows platform teams to onboard new tenants via pull requests.

```bash
#!/bin/bash
# scripts/onboard-tenant.sh
# Usage: ./onboard-tenant.sh tenant-b https://github.com/my-org/tenant-b-app

set -euo pipefail

TENANT_ID="${1:?Tenant ID required}"
TENANT_REPO="${2:?Tenant Git repository URL required}"
BASE_DIR="infrastructure/tenants"
TENANT_DIR="${BASE_DIR}/${TENANT_ID}"

# Create tenant directory structure
mkdir -p "${TENANT_DIR}/image-automation"

# Generate namespace and RBAC from template
sed "s/tenant-a/${TENANT_ID}/g" \
  "${BASE_DIR}/tenant-a/namespace.yaml" \
  > "${TENANT_DIR}/namespace.yaml"

# Generate GitRepository
cat > "${TENANT_DIR}/gitrepository-${TENANT_ID}.yaml" << EOF

  name: ${TENANT_ID}-app

cat > "${BASE_DIR}/../platform/kustomization-${TENANT_ID}-workloads.yaml" << EOF

  name: ${TENANT_ID}-workloads

    - name: platform-infra
      namespace: flux-system

echo "Tenant ${TENANT_ID} onboarding manifests generated."
```

## Security Hardening

### Preventing Cross-Namespace Source References Without Explicit Allow

By default, Flux allows cross-namespace references in Kustomizations and HelmReleases. This should be restricted so tenants cannot reference Sources in flux-system.

```yaml
# clusters/production/platform/kustomize-controller-patch.yaml
# Patch the kustomize-controller to restrict cross-namespace references
apiVersion: apps/v1
kind: Deployment
metadata:
  name: kustomize-controller
  namespace: flux-system
spec:
  template:
    spec:
      containers:
        - name: manager
          args:
            - --watch-all-namespaces=true
            # Disable cross-namespace references from tenant namespaces
            # Tenants must use Sources in their own namespace
            - --no-cross-namespace-refs=false
```

### Admission Policy to Block Privileged Containers

```yaml
# infrastructure/policies/no-privileged-containers.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-privileged-containers
spec:
  validationFailureAction: Enforce
  background: true
  rules:
    - name: check-privileged
      match:
        any:
          - resources:
              kinds: ["Pod"]
              namespaceSelector:
                matchLabels:
                  tenant: "true"
      validate:
        message: "Privileged containers are not allowed in tenant namespaces."
        pattern:
          spec:
            containers:
              - =(securityContext):
                  =(privileged): "false"
```

## Summary

A robust Flux multi-tenancy implementation requires coordination across multiple layers. The platform team bootstraps Flux with image automation controllers, defines shared infrastructure via cluster-scoped Kustomizations, and provisions tenant namespaces with appropriately scoped ServiceAccounts. Tenant sources (GitRepository, OCIRepository) live in tenant namespaces, and tenant Kustomizations reference them via cross-namespace sourceRef while enforcing isolation through ServiceAccount impersonation. Image automation closes the GitOps loop by automatically updating image tags in tenant repositories, and the notification controller ensures all stakeholders receive timely delivery status. Multi-cluster push mode extends this model across multiple target clusters from a single management cluster, maintaining the same isolation boundaries.
