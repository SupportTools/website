Dependency injection in Go provokes strong opinions. Reflection-based DI containers (like those common in Java/C# ecosystems) feel un-idiomatic because they move errors from compile time to runtime, obscure dependencies, and add overhead. Wire takes a different approach: it generates the dependency wiring code at compile time, producing plain Go code that you can read, debug, and understand. This guide covers Wire's advanced patterns beyond the getting-started tutorials.

# Go Compile-Time Dependency Injection: Wire Advanced Patterns

## Why Wire Over Manual Wiring

Manual dependency injection in Go is a `main.go` or `server.go` that manually constructs and wires all components:

```go
// Manual wiring — works, but becomes unwieldy at scale
func main() {
    db, err := database.New(cfg.DSN)
    if err != nil {
        log.Fatal(err)
    }

    userRepo := repository.NewUser(db)
    sessionRepo := repository.NewSession(db)
    emailService := email.NewSMTP(cfg.SMTPHost)
    userService := service.NewUser(userRepo, sessionRepo, emailService)
    authService := service.NewAuth(userRepo, sessionRepo, cfg.JWTSecret)
    userHandler := handler.NewUser(userService, authService)
    authHandler := handler.NewAuth(authService)
    // ... 50 more lines
}
```

Wire generates this boilerplate from provider functions you write, catching dependency problems at compile time. If you forget to provide a dependency, the build fails with a clear error.

## Section 2: Provider Functions

A provider is any Go function that returns a value (and optionally an error and cleanup function).

### Basic Provider

```go
// internal/database/database.go
package database

import (
    "database/sql"
    "fmt"

    _ "github.com/lib/pq"
)

// Config holds database connection parameters.
type Config struct {
    DSN             string
    MaxOpenConns    int
    MaxIdleConns    int
    ConnMaxLifetime time.Duration
}

// New creates a new database connection.
// Wire will call this function when something depends on *sql.DB.
func New(cfg Config) (*sql.DB, func(), error) {
    db, err := sql.Open("postgres", cfg.DSN)
    if err != nil {
        return nil, nil, fmt.Errorf("opening database: %w", err)
    }

    db.SetMaxOpenConns(cfg.MaxOpenConns)
    db.SetMaxIdleConns(cfg.MaxIdleConns)
    db.SetConnMaxLifetime(cfg.ConnMaxLifetime)

    if err := db.Ping(); err != nil {
        db.Close()
        return nil, nil, fmt.Errorf("pinging database: %w", err)
    }

    cleanup := func() {
        db.Close()
    }

    return db, cleanup, nil
}
```

### Struct Provider

When all struct fields are exported and match provider outputs by type, use `wire.Struct`:

```go
// internal/service/user.go
package service

import "github.com/myapp/internal/repository"

// UserService handles user business logic.
type UserService struct {
    Users    *repository.UserRepository
    Sessions *repository.SessionRepository
    Email    EmailSender
    Config   UserConfig
}

// NewUser creates a UserService.
// Wire can also generate this with wire.Struct(&UserService{}, "*")
func NewUser(
    users *repository.UserRepository,
    sessions *repository.SessionRepository,
    email EmailSender,
    cfg UserConfig,
) *UserService {
    return &UserService{
        Users:    users,
        Sessions: sessions,
        Email:    email,
        Config:   cfg,
    }
}
```

## Section 3: Provider Sets

Provider sets group related providers together. They are the primary reuse mechanism in Wire.

```go
// internal/app/providers.go
package app

import (
    "github.com/google/wire"
    "github.com/myapp/internal/database"
    "github.com/myapp/internal/repository"
    "github.com/myapp/internal/service"
    "github.com/myapp/internal/handler"
    "github.com/myapp/internal/email"
)

// DatabaseSet provides database infrastructure.
var DatabaseSet = wire.NewSet(
    database.New,
)

// RepositorySet provides all repositories.
// Depends on: *sql.DB (from DatabaseSet)
var RepositorySet = wire.NewSet(
    repository.NewUser,
    repository.NewSession,
    repository.NewProduct,
)

// EmailSet provides email infrastructure.
var EmailSet = wire.NewSet(
    email.NewSMTPSender,
    ProvideEmailConfig,  // Local provider function
)

// ServiceSet provides all services.
// Depends on: repositories, email
var ServiceSet = wire.NewSet(
    service.NewUser,
    service.NewAuth,
    service.NewProduct,
)

// HandlerSet provides all HTTP handlers.
var HandlerSet = wire.NewSet(
    handler.NewUser,
    handler.NewAuth,
    handler.NewProduct,
)

// AppSet is the top-level set for the entire application.
var AppSet = wire.NewSet(
    DatabaseSet,
    RepositorySet,
    EmailSet,
    ServiceSet,
    HandlerSet,
)
```

### Nested Provider Sets

Provider sets can include other provider sets. Wire flattens the graph at code generation time.

```go
// Infrastructure layer
var InfraSet = wire.NewSet(
    DatabaseSet,
    EmailSet,
    CacheSet,
)

// Domain layer
var DomainSet = wire.NewSet(
    RepositorySet,
    ServiceSet,
)

// Presentation layer
var PresentationSet = wire.NewSet(
    HandlerSet,
    MiddlewareSet,
)

// Full application
var AppSet = wire.NewSet(
    InfraSet,
    DomainSet,
    PresentationSet,
)
```

## Section 4: Interface Binding

Wire does not know which concrete type to use for an interface unless you tell it. `wire.Bind` handles this.

```go
// internal/email/interfaces.go
package email

// Sender is the interface for sending emails.
type Sender interface {
    SendWelcome(to, username string) error
    SendPasswordReset(to, token string) error
}
```

```go
// internal/email/smtp.go
package email

// SMTPSender implements Sender via SMTP.
type SMTPSender struct {
    host     string
    port     int
    from     string
    password string
}

func NewSMTPSender(cfg SMTPConfig) *SMTPSender {
    return &SMTPSender{
        host:     cfg.Host,
        port:     cfg.Port,
        from:     cfg.From,
        password: cfg.Password,
    }
}

func (s *SMTPSender) SendWelcome(to, username string) error { /* ... */ return nil }
func (s *SMTPSender) SendPasswordReset(to, token string) error { /* ... */ return nil }
```

```go
// internal/app/providers.go — bind interface to implementation
var EmailSet = wire.NewSet(
    email.NewSMTPSender,
    // Tell Wire: when something needs email.Sender, provide *email.SMTPSender
    wire.Bind(new(email.Sender), new(*email.SMTPSender)),
)
```

### Multiple Implementations of the Same Interface

When you need to provide different implementations (e.g., real vs. mock):

```go
// Separate provider sets for different implementations
var ProductionEmailSet = wire.NewSet(
    email.NewSMTPSender,
    wire.Bind(new(email.Sender), new(*email.SMTPSender)),
)

var DevelopmentEmailSet = wire.NewSet(
    email.NewLogSender,  // Logs emails instead of sending
    wire.Bind(new(email.Sender), new(*email.LogSender)),
)
```

## Section 5: Cleanup Functions

Wire tracks cleanup functions and chains them together. The generated `cleanup` function calls them in reverse order of creation (LIFO).

```go
// Provider returning a cleanup function
func NewRedisClient(cfg RedisConfig) (*redis.Client, func(), error) {
    client := redis.NewClient(&redis.Options{
        Addr:     cfg.Addr,
        Password: cfg.Password,
        DB:       cfg.DB,
    })

    if err := client.Ping(context.Background()).Err(); err != nil {
        return nil, nil, fmt.Errorf("pinging Redis: %w", err)
    }

    cleanup := func() {
        if err := client.Close(); err != nil {
            log.Printf("closing Redis client: %v", err)
        }
    }

    return client, cleanup, nil
}
```

Wire generates code like:

```go
// wire_gen.go (generated)
func InitializeApp(cfg Config) (*App, func(), error) {
    db, cleanup, err := database.New(cfg.Database)
    if err != nil {
        return nil, nil, err
    }

    redisClient, cleanup2, err := NewRedisClient(cfg.Redis)
    if err != nil {
        cleanup()  // Wire chains cleanups automatically
        return nil, nil, err
    }

    // ... more initialization

    app := &App{/* ... */}

    return app, func() {
        cleanup2()  // Redis closed first
        cleanup()   // DB closed second
    }, nil
}
```

## Section 6: Optional Bindings

Optional bindings allow a dependency to be nil if no provider is available. Use this for optional features or debugging tools.

```go
// Define an optional interface
type Tracer interface {
    StartSpan(name string) (Span, func())
}

// NilTracer is a no-op implementation
type NilTracer struct{}
func (n *NilTracer) StartSpan(name string) (Span, func()) { return NilSpan{}, func(){} }

// Wire injector with optional tracer
// wire.go
//go:build wireinject
// +build wireinject

package main

import "github.com/google/wire"

func InitializeApp(cfg Config) (*App, func(), error) {
    wire.Build(
        AppSet,
        // Provide the nil tracer as default if no real tracer is configured
        wire.Value((*NilTracer)(nil)),
        wire.Bind(new(Tracer), new(*NilTracer)),
    )
    return nil, nil, nil
}

// For production with real tracer (separate injector)
func InitializeAppWithTracing(cfg Config) (*App, func(), error) {
    wire.Build(
        AppSet,
        jaeger.NewTracer,
        wire.Bind(new(Tracer), new(*jaeger.Tracer)),
    )
    return nil, nil, nil
}
```

## Section 7: Value Providers

`wire.Value` provides a constant or pre-computed value without a provider function:

```go
func InitializeApp(cfg Config) (*App, func(), error) {
    wire.Build(
        AppSet,
        // Inject a specific string value
        wire.Value("production"),
        // Inject a pre-built struct
        wire.Value(http.DefaultClient),
        // Inject an interface value
        wire.InterfaceValue(new(io.Writer), os.Stdout),
    )
    return nil, nil, nil
}
```

## Section 8: Wire Injectors

The injector is the function Wire generates wiring code for. It must have the `//go:build wireinject` constraint and call `wire.Build`.

```go
// cmd/server/wire.go
//go:build wireinject
// +build wireinject

package main

import (
    "github.com/google/wire"
    "github.com/myapp/internal/app"
    "github.com/myapp/internal/config"
)

// InitializeServer creates the fully initialized HTTP server.
// Wire will generate the implementation in wire_gen.go.
func InitializeServer(cfg *config.Config) (*http.Server, func(), error) {
    wire.Build(
        app.AppSet,
        // Provide sub-configs extracted from the top-level config
        ProvideDBConfig,
        ProvideEmailConfig,
        ProvideHTTPConfig,
    )
    return nil, nil, nil
}

// ProvideDBConfig extracts database config from the top-level config.
func ProvideDBConfig(cfg *config.Config) database.Config {
    return database.Config{
        DSN:             cfg.DatabaseDSN,
        MaxOpenConns:    cfg.DBMaxOpenConns,
        MaxIdleConns:    cfg.DBMaxIdleConns,
        ConnMaxLifetime: cfg.DBConnMaxLifetime,
    }
}

func ProvideEmailConfig(cfg *config.Config) email.SMTPConfig {
    return email.SMTPConfig{
        Host:     cfg.SMTPHost,
        Port:     cfg.SMTPPort,
        From:     cfg.EmailFrom,
        Password: cfg.SMTPPassword,
    }
}
```

## Section 9: Testing with Wire Injectors

Wire's real power for testing is that you can create test-specific injectors that substitute mock implementations.

### Test Injector

```go
// internal/service/user_test.go
//go:build wireinject
// +build wireinject

package service_test

import (
    "testing"

    "github.com/google/wire"
    "github.com/myapp/internal/repository"
    "github.com/myapp/internal/service"
    "github.com/myapp/internal/email"
)

// TestProviderSet provides test doubles.
var TestProviderSet = wire.NewSet(
    repository.NewInMemoryUserRepository,   // In-memory implementation
    repository.NewInMemorySessionRepository,
    email.NewMockSender,                    // Mock that captures sent emails
    wire.Bind(new(email.Sender), new(*email.MockSender)),
)

// InitializeUserServiceForTest creates a UserService with test doubles.
func InitializeUserServiceForTest(t *testing.T) (*service.UserService, *email.MockSender) {
    wire.Build(
        TestProviderSet,
        service.NewUser,
        ProvideTestUserConfig,
    )
    return nil, nil
}

func ProvideTestUserConfig(t *testing.T) service.UserConfig {
    return service.UserConfig{
        MaxLoginAttempts: 5,
        SessionTTL:       time.Hour,
    }
}
```

```go
// email/mock.go
package email

import "sync"

// MockSender is a test double that captures sent emails.
type MockSender struct {
    mu     sync.Mutex
    Sent   []SentEmail
}

type SentEmail struct {
    To      string
    Subject string
    Body    string
}

func NewMockSender() *MockSender {
    return &MockSender{}
}

func (m *MockSender) SendWelcome(to, username string) error {
    m.mu.Lock()
    defer m.mu.Unlock()
    m.Sent = append(m.Sent, SentEmail{
        To:      to,
        Subject: "Welcome",
        Body:    "Welcome " + username,
    })
    return nil
}

func (m *MockSender) SentCount() int {
    m.mu.Lock()
    defer m.mu.Unlock()
    return len(m.Sent)
}

func (m *MockSender) LastSentTo() string {
    m.mu.Lock()
    defer m.mu.Unlock()
    if len(m.Sent) == 0 {
        return ""
    }
    return m.Sent[len(m.Sent)-1].To
}
```

### Using Test Injectors

```go
// internal/service/user_test.go (non-wire portion)
//go:build !wireinject
// +build !wireinject

package service_test

import (
    "testing"

    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func TestUserService_RegisterNewUser(t *testing.T) {
    svc, mockEmail := InitializeUserServiceForTest(t)

    err := svc.Register("alice@example.com", "alice", "password123")
    require.NoError(t, err)

    // Verify the welcome email was sent
    assert.Equal(t, 1, mockEmail.SentCount())
    assert.Equal(t, "alice@example.com", mockEmail.LastSentTo())
}

func TestUserService_DuplicateEmail(t *testing.T) {
    svc, _ := InitializeUserServiceForTest(t)

    err := svc.Register("alice@example.com", "alice", "password123")
    require.NoError(t, err)

    err = svc.Register("alice@example.com", "alice2", "password456")
    assert.ErrorIs(t, err, service.ErrEmailTaken)
}
```

## Section 10: Advanced Patterns

### Parameterized Providers

When a provider needs a parameter that is not a dependency (e.g., an environment name):

```go
// ProvideLogLevel returns the log level based on environment.
func ProvideLogLevel(env string) slog.Level {
    switch env {
    case "production":
        return slog.LevelWarn
    case "development":
        return slog.LevelDebug
    default:
        return slog.LevelInfo
    }
}

// Inject the environment string as a value
func InitializeApp(env string, cfg *Config) (*App, func(), error) {
    wire.Build(
        AppSet,
        wire.Value(env),  // env string is now available to providers
        ProvideLogLevel,
    )
    return nil, nil, nil
}
```

### Wire with Context

```go
// Context-aware provider
func NewHTTPServer(ctx context.Context, cfg ServerConfig, handler http.Handler) (*http.Server, func(), error) {
    srv := &http.Server{
        Addr:    cfg.Addr,
        Handler: handler,
    }

    cleanup := func() {
        shutdownCtx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
        defer cancel()
        if err := srv.Shutdown(shutdownCtx); err != nil {
            log.Printf("server shutdown: %v", err)
        }
    }

    return srv, cleanup, nil
}
```

### Error Handling in Wire

Wire supports providers that return errors. The generated code propagates errors upward and calls cleanups for already-initialized resources:

```go
// If database.New returns an error, Wire-generated code will:
// 1. Not call any downstream providers
// 2. Return the error from InitializeApp
// 3. Call cleanup functions for any already-initialized resources

func InitializeApp(cfg Config) (*App, func(), error) {
    wire.Build(AppSet)
    return nil, nil, nil  // Wire replaces this
}

// Generated wire_gen.go:
func InitializeApp(cfg Config) (*App, func(), error) {
    db, cleanup, err := database.New(cfg.Database)
    if err != nil {
        return nil, nil, err  // Error propagated; no cleanups to call yet
    }
    redisClient, cleanup2, err := NewRedisClient(cfg.Redis)
    if err != nil {
        cleanup()  // DB cleanup called since it was already initialized
        return nil, nil, err
    }
    // ...
}
```

## Section 11: Common Wire Errors and Solutions

```
# Error: "missing provider for X"
# Solution: Add a provider for X to the wire.Build call or to a ProviderSet used in it

# Error: "ambiguous binding for interface Y"
# Solution: Use wire.Bind explicitly to specify which concrete type to use

# Error: "provider set cycle"
# Solution: A depends on B which depends on A — redesign to break the cycle

# Error: "provider has no output for type X"
# Solution: The provider function's return types don't match what's needed downstream
```

## Conclusion

Wire's compile-time approach to dependency injection catches wiring errors before deployment, produces readable generated code, and scales cleanly from small services to large monorepos. The key patterns — provider sets for reuse, interface binding for flexibility, cleanup function chaining for resource management, and test-specific injectors for isolated unit tests — address the real challenges of dependency injection at scale. By generating plain Go code rather than using reflection, Wire stays true to Go's design philosophy while delivering the benefits of structured dependency management.
