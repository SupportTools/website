Software supply chain attacks — SolarWinds, XZ Utils, log4shell — have fundamentally changed how enterprises think about the integrity of software they build and deploy. The question is no longer just "is this code safe?" but "can we prove where this artifact came from, how it was built, and that it hasn't been tampered with?" Tekton Chains answers this by automatically intercepting Tekton PipelineRun completions and generating cryptographically signed provenance attestations conforming to the SLSA (Supply-chain Levels for Software Artifacts) framework. Combined with Sigstore's keyless signing infrastructure and Kyverno admission policies, this creates a verifiable chain of custody from source code to running container.

# Kubernetes Tekton Chains: Supply Chain Security

## Supply Chain Security Fundamentals

Before diving into Tekton Chains, the concepts:

**SLSA (Supply-chain Levels for Software Artifacts)**: A framework that defines four levels of supply chain integrity. SLSA Level 1 requires provenance documentation. Level 2 requires signed provenance generated by a hosted build platform. Level 3 requires that the build platform itself is hardened and the provenance cannot be forged even by the build process.

**Provenance**: A verifiable record of how an artifact was produced. SLSA provenance includes: the builder identity, build invocation (inputs, parameters), build environment, and the resulting artifact(s) with their digests.

**Attestation**: A signed statement about a subject (typically an OCI artifact identified by digest). Attestations are stored in OCI registries alongside the artifact they describe, using the OCI referrers API.

**Cosign**: Part of the Sigstore project. Signs container images and stores signatures/attestations in OCI registries. Supports keyless signing using OIDC identity tokens (no long-lived private keys stored anywhere).

**Sigstore's Rekor**: An append-only transparency log for signatures. Keyless cosign signatures are logged to Rekor, providing a public audit trail.

**SBOM (Software Bill of Materials)**: A machine-readable inventory of all components in a software artifact. Attestations can contain SBOMs in SPDX or CycloneDX format.

## Architecture Overview

```
Developer pushes to Git
        │
        ▼
Tekton EventListener (Triggers)
        │
        ▼
Tekton Pipeline runs:
  ├── Clone source
  ├── Build container image (kaniko)
  ├── Run tests
  └── Push to registry
        │
        ▼ (PipelineRun completes)
Tekton Chains intercepts
        │
        ▼
Chains generates SLSA provenance attestation
        │
        ├── Signs with Sigstore keyless (workload identity)
        │         │
        │         └── Certificate logged to Fulcio CA
        │         └── Signature logged to Rekor transparency log
        │
        └── Stores attestation in OCI registry
              (as referrer to the image)
        │
        ▼
Kyverno admission policy:
  "Only deploy images with valid Chains provenance"
        │
        ▼
Runtime: Image runs in cluster only if policy passes
```

## Configuring Tekton Chains

```yaml
# tekton-chains-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: chains-config
  namespace: tekton-chains
data:
  # Artifact formats
  # "slsav1" generates SLSA v1.0 Build L2 provenance
  artifacts.taskrun.format: "slsav1"
  artifacts.pipelinerun.format: "slsav1"

  # Storage backends for attestations
  # "oci" stores attestations in the OCI registry alongside the artifact
  artifacts.taskrun.storage: "oci"
  artifacts.pipelinerun.storage: "oci,tekton"
  # "tekton" also stores in a Kubernetes Secret for debugging

  # Signature format
  # "sigstore-bundle" creates a Sigstore bundle (signature + certificate + transparency log entry)
  signatures-backend: "x509"

  # Transparency log
  transparency.enabled: "true"
  transparency.url: "https://rekor.sigstore.dev"

  # For keyless signing with workload identity (recommended):
  # fulcio.address: "https://fulcio.sigstore.dev"
  # The service account's OIDC token is used for identity

  # OCI registry configuration
  storage.oci.repository: ""  # Empty = same registry as the image
```

Apply the configuration:

```bash
kubectl apply -f tekton-chains-config.yaml
```

## Signing Keys: Keyless vs. KMS

### Option 1: Keyless Signing with Workload Identity (Recommended)

Keyless signing uses the Chains controller's Kubernetes service account to obtain an OIDC token, which Sigstore's Fulcio CA uses to issue a short-lived certificate. No private key needs to be stored or managed.

```bash
# Enable the Chains service account for workload identity (GKE example)
kubectl annotate serviceaccount tekton-chains-controller \
  -n tekton-chains \
  iam.gke.io/gcp-service-account=tekton-chains@<project>.iam.gserviceaccount.com

# Grant the service account permission to sign (OIDC token issuance)
gcloud iam service-accounts add-iam-policy-binding \
  tekton-chains@<project>.iam.gserviceaccount.com \
  --role=roles/iam.workloadIdentityUser \
  --member="serviceAccount:<project>.svc.id.goog[tekton-chains/tekton-chains-controller]"
```

For EKS with IRSA:
```bash
# Annotate the service account for AWS IRSA
kubectl annotate serviceaccount tekton-chains-controller \
  -n tekton-chains \
  eks.amazonaws.com/role-arn=arn:aws:iam::<account>:role/tekton-chains

# The Chains controller can use this identity with Sigstore
```

### Option 2: KMS-Based Signing

For air-gapped environments or where keyless signing is not available:

```bash
# Generate a signing key pair and store in KMS (AWS KMS example)
# First, create the key:
aws kms create-key \
  --description "Tekton Chains signing key" \
  --key-usage SIGN_VERIFY \
  --key-spec ECC_NIST_P256

# Configure Chains to use KMS
kubectl patch configmap chains-config \
  -n tekton-chains \
  --type=merge \
  -p '{"data":{"signers.kms.kmsref":"awskms:///arn:aws:kms:<region>:<account>:key/<key-id>"}}'
```

### Option 3: Local Key (Development/Air-gapped)

```bash
# Generate a cosign key pair (not recommended for production)
cosign generate-key-pair k8s://tekton-chains/signing-secrets

# This stores the private key as a Kubernetes Secret:
# tekton-chains/signing-secrets
```

## Writing a SLSA-Compliant Tekton Pipeline

Chains automatically captures provenance from pipeline runs, but you need to properly annotate steps so Chains knows which output is the artifact to attest:

```yaml
# pipeline-build.yaml
apiVersion: tekton.dev/v1
kind: Pipeline
metadata:
  name: build-and-attest
  namespace: ci
spec:
  params:
  - name: git-url
    type: string
  - name: git-revision
    type: string
  - name: image-name
    type: string
  - name: image-registry
    type: string

  workspaces:
  - name: source
  - name: docker-credentials

  results:
  - name: IMAGE_DIGEST
    description: The digest of the built image
    value: $(tasks.build-image.results.IMAGE_DIGEST)
  - name: IMAGE_URL
    description: The URL of the built image
    value: $(tasks.build-image.results.IMAGE_URL)

  tasks:
  - name: fetch-source
    taskRef:
      name: git-clone
      kind: ClusterTask
    params:
    - name: url
      value: $(params.git-url)
    - name: revision
      value: $(params.git-revision)
    workspaces:
    - name: output
      workspace: source

  - name: run-tests
    runAfter: [fetch-source]
    taskRef:
      name: run-tests
    workspaces:
    - name: source
      workspace: source

  - name: build-image
    runAfter: [run-tests]
    taskRef:
      name: kaniko-build
    params:
    - name: IMAGE
      value: $(params.image-registry)/$(params.image-name)
    - name: DOCKERFILE
      value: ./Dockerfile
    workspaces:
    - name: source
      workspace: source
    - name: dockerconfig
      workspace: docker-credentials

---
# The kaniko build task must output IMAGE_URL and IMAGE_DIGEST
# for Chains to know which artifact to attest
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: kaniko-build
  namespace: ci
  annotations:
    # Tell Chains that the IMAGE result is the artifact
    chains.tekton.dev/scan-result-image-url: $(results.IMAGE_URL.path)
    chains.tekton.dev/scan-result-image-digest: $(results.IMAGE_DIGEST.path)
spec:
  params:
  - name: IMAGE
    type: string
  - name: DOCKERFILE
    type: string
    default: ./Dockerfile
  - name: CONTEXT
    type: string
    default: .

  workspaces:
  - name: source
  - name: dockerconfig
    optional: true

  results:
  - name: IMAGE_URL
    description: "The URL of the built image"
  - name: IMAGE_DIGEST
    description: "The digest of the built image (sha256:...)"

  steps:
  - name: build-and-push
    image: gcr.io/kaniko-project/executor:v1.19.2
    args:
    - --dockerfile=$(params.DOCKERFILE)
    - --context=dir://$(workspaces.source.path)/$(params.CONTEXT)
    - --destination=$(params.IMAGE)
    - --digest-file=$(results.IMAGE_DIGEST.path)
    - --image-name-with-digest-file=$(results.IMAGE_URL.path)
    - --cache=true
    - --cache-ttl=24h
    # Security: run as non-root with minimal capabilities
    securityContext:
      runAsNonRoot: false  # kaniko requires root for overlay filesystem
      allowPrivilegeEscalation: false
    volumeMounts:
    - name: docker-config
      mountPath: /kaniko/.docker
    resources:
      requests:
        cpu: "500m"
        memory: "1Gi"
      limits:
        cpu: "2"
        memory: "4Gi"

  volumes:
  - name: docker-config
    projected:
      sources:
      - secret:
          name: registry-credentials
          items:
          - key: .dockerconfigjson
            path: config.json
```

## Triggering Pipelines on Git Push

```yaml
# tekton-triggers.yaml
---
apiVersion: triggers.tekton.dev/v1beta1
kind: EventListener
metadata:
  name: github-listener
  namespace: ci
spec:
  serviceAccountName: tekton-triggers-sa
  triggers:
  - name: github-push
    interceptors:
    - ref:
        name: github
      params:
      - name: eventTypes
        value: ["push"]
      - name: secretRef
        value:
          secretName: github-webhook-secret
          secretKey: token
    bindings:
    - ref: github-push-binding
    template:
      ref: build-pipeline-template

---
apiVersion: triggers.tekton.dev/v1beta1
kind: TriggerBinding
metadata:
  name: github-push-binding
  namespace: ci
spec:
  params:
  - name: git-url
    value: $(body.repository.clone_url)
  - name: git-revision
    value: $(body.head_commit.id)
  - name: image-name
    value: $(body.repository.name)

---
apiVersion: triggers.tekton.dev/v1beta1
kind: TriggerTemplate
metadata:
  name: build-pipeline-template
  namespace: ci
spec:
  params:
  - name: git-url
  - name: git-revision
  - name: image-name

  resourcetemplates:
  - apiVersion: tekton.dev/v1
    kind: PipelineRun
    metadata:
      generateName: build-$(tt.params.image-name)-
      namespace: ci
      annotations:
        # Chains reads these to set provenance metadata
        chains.tekton.dev/signed: "true"
    spec:
      pipelineRef:
        name: build-and-attest
      params:
      - name: git-url
        value: $(tt.params.git-url)
      - name: git-revision
        value: $(tt.params.git-revision)
      - name: image-name
        value: $(tt.params.image-name)
      - name: image-registry
        value: registry.example.com/builds
      workspaces:
      - name: source
        volumeClaimTemplate:
          spec:
            accessModes: [ReadWriteOnce]
            resources:
              requests:
                storage: 1Gi
      - name: docker-credentials
        secret:
          secretName: registry-credentials
```

## Verifying Provenance After Build

After a pipeline run completes, Chains generates and signs a provenance attestation. Verify it:

```bash
# Install cosign
curl -O https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64
chmod +x cosign-linux-amd64
mv cosign-linux-amd64 /usr/local/bin/cosign

# The image reference (including digest is important for verification)
IMAGE="registry.example.com/builds/my-app@sha256:<digest>"

# Verify the image signature
cosign verify \
  --certificate-identity-regexp "https://tekton.dev/.*" \
  --certificate-oidc-issuer "https://kubernetes.default.svc.cluster.local" \
  "$IMAGE"

# For keyless signing with Sigstore public good instance:
cosign verify \
  --certificate-identity-regexp ".*" \
  --certificate-oidc-issuer "https://accounts.google.com" \
  "$IMAGE"

# Download and inspect the SLSA provenance attestation
cosign download attestation "$IMAGE" | \
  jq -r '.payload' | base64 -d | jq .
```

Example provenance attestation (SLSA v1.0):

## Verifying Attestation Signatures with cosign

```bash
# Verify the SLSA provenance attestation specifically
cosign verify-attestation \
  --type slsaprovenance1 \
  --certificate-identity-regexp "https://tekton.dev/.*" \
  --certificate-oidc-issuer "https://kubernetes.default.svc.cluster.local" \
  "$IMAGE" | jq -r '.payload' | base64 -d | jq '.predicate'

# For KMS-signed attestations
cosign verify-attestation \
  --type slsaprovenance1 \
  --key awskms:///arn:aws:kms:<region>:<account>:key/<key-id> \
  "$IMAGE"

# Verify against Rekor transparency log
cosign verify \
  --rekor-url https://rekor.sigstore.dev \
  "$IMAGE"

# Check the Rekor log entry
COSIGN_EXPERIMENTAL=1 cosign triangulate "$IMAGE"
# Returns the rekor log URL for this artifact
```

## Policy Enforcement with Kyverno

Kyverno admission policies enforce that only images with valid Chains provenance can be deployed:

```yaml
# kyverno-policy-require-provenance.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-slsa-provenance
  annotations:
    policies.kyverno.io/title: Require SLSA Provenance
    policies.kyverno.io/category: Supply Chain Security
    policies.kyverno.io/severity: high
    policies.kyverno.io/description: >-
      Requires all containers to have a valid SLSA provenance attestation
      signed by the Tekton Chains controller.
spec:
  validationFailureAction: Enforce
  background: false
  rules:
  - name: check-slsa-provenance
    match:
      any:
      - resources:
          kinds:
          - Pod
          namespaceSelector:
            matchLabels:
              supply-chain-enforcement: enabled
    verifyImages:
    - imageReferences:
      - "registry.example.com/builds/*"
      attestors:
      - count: 1
        entries:
        - keyless:
            subject: "https://tekton.dev/*"
            issuer: "https://kubernetes.default.svc.cluster.local"
            rekor:
              url: https://rekor.sigstore.dev
      attestations:
      - predicateType: https://slsa.dev/provenance/v1
        conditions:
        - all:
          # Require the build was performed by our Tekton instance
          - key: "{{ predicate.runDetails.builder.id }}"
            operator: AnyIn
            value:
            - "https://tekton.dev/chains/v2@*"
          # Require the source came from our GitHub org
          - key: "{{ predicate.buildDefinition.externalParameters.runSpec.params[?name=='git-url'].value | [0] }}"
            operator: AnyIn
            value:
            - "https://github.com/yourorg/*"
```

```yaml
# For KMS-signed images
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-chains-signature-kms
spec:
  validationFailureAction: Enforce
  rules:
  - name: verify-image-signature
    match:
      any:
      - resources:
          kinds: [Pod]
    verifyImages:
    - imageReferences:
      - "registry.example.com/builds/*"
      attestors:
      - entries:
        - keys:
            kms: awskms:///arn:aws:kms:<region>:<account>:key/<key-id>
            rekor:
              url: https://rekor.sigstore.dev
```

Enable enforcement in specific namespaces:

```bash

kubectl label namespace staging supply-chain-enforcement=enabled
```

## SBOM Generation and Attestation

Add SBOM generation to your pipeline with Syft:

```yaml
# task-generate-sbom.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: generate-sbom
  namespace: ci
spec:
  params:
  - name: IMAGE
    type: string
  - name: IMAGE_DIGEST
    type: string

  results:
  - name: SBOM_PATH
    description: "Path to the generated SBOM"

  steps:
  - name: generate-syft-sbom
    image: anchore/syft:v0.98.0
    args:
    - scan
    - $(params.IMAGE)@$(params.IMAGE_DIGEST)
    - --output
    - spdx-json=/workspace/sbom.spdx.json
    - --output
    - cyclonedx-json=/workspace/sbom.cdx.json
    resources:
      requests:
        cpu: "200m"
        memory: "512Mi"

  - name: attest-sbom
    image: gcr.io/projectsigstore/cosign:v2.2.3
    script: |
      #!/bin/bash
      set -euo pipefail

      # Attach the SBOM as an attestation
      cosign attest \
        --predicate /workspace/sbom.spdx.json \
        --type spdxjson \
        --yes \
        $(params.IMAGE)@$(params.IMAGE_DIGEST)

      echo -n "/workspace/sbom.spdx.json" > $(results.SBOM_PATH.path)
    env:
    - name: COSIGN_EXPERIMENTAL
      value: "1"
```

Verify the SBOM attestation:

```bash
cosign verify-attestation \
  --type spdxjson \
  --certificate-identity-regexp ".*" \
  --certificate-oidc-issuer "https://kubernetes.default.svc.cluster.local" \
  "registry.example.com/builds/my-app@sha256:<digest>" | \
  jq -r '.payload' | base64 -d | jq '.predicate.packages[].name'
```

## Debugging Tekton Chains

```bash
# Check Chains controller logs
kubectl logs -n tekton-chains \
  deployment/tekton-chains-controller \
  --tail=100 | grep -E "(ERROR|WARN|signed|attestation)"

# Check the signing status on a completed PipelineRun
PIPELINE_RUN_NAME="build-my-app-abc12"
kubectl get pipelinerun "$PIPELINE_RUN_NAME" -n ci \
  -o jsonpath='{.metadata.annotations}' | jq .

# Look for these annotations added by Chains:
# chains.tekton.dev/signed: "true"
# chains.tekton.dev/transparency-url: "https://rekor.sigstore.dev/..."

# If signing failed, check:
kubectl get pipelinerun "$PIPELINE_RUN_NAME" -n ci \
  -o jsonpath='{.metadata.annotations.chains\.tekton\.dev/signing-summary}'

# Check that the OCI storage write succeeded
cosign download attestation \
  "registry.example.com/builds/my-app@sha256:<digest>" 2>&1 | head -20

# View the TaskRun results captured by Chains
kubectl get taskrun -n ci \
  -l tekton.dev/pipeline="$PIPELINE_RUN_NAME" \
  -o json | jq '.items[].status.taskResults'
```

## SLSA Level Assessment

After implementing Tekton Chains, assess your SLSA level:

**SLSA Level 1 (Build L1)**:
- Build process is scripted (Tekton Pipeline) ✓
- Provenance is generated ✓

**SLSA Level 2 (Build L2)**:
- Build runs on a hosted platform (Kubernetes) ✓
- Provenance is signed by the build platform (Chains) ✓
- Provenance is stored in an OCI registry (Chains) ✓

**SLSA Level 3 (Build L3)**:
- Build platform is hardened (requires additional controls)
- Build environment is ephemeral (Tekton pods are ephemeral) ✓
- Builds are isolated (separate pods per PipelineRun) ✓
- Provenance cannot be forged by the build process (requires no shared secrets between builds)

Requirements for Level 3 that need additional work:
- The Chains signing key must not be accessible to the build tasks themselves.
- Network isolation between build tasks and the signing infrastructure.
- Immutable build infrastructure (GitOps-managed Tekton installation).

## Conclusion

Tekton Chains provides SLSA-compliant supply chain security with minimal additional pipeline complexity. Once installed and configured, it automatically intercepts every pipeline run completion and generates cryptographically signed provenance attestations without any changes to existing pipelines — provided pipelines export the `IMAGE_URL` and `IMAGE_DIGEST` results that Chains uses to identify the artifact. The combination of Chains for provenance generation, Sigstore/Cosign for keyless signing, and Kyverno for admission policy enforcement creates a complete, auditable supply chain where every running container in the cluster has a verifiable history: which source commit was used, which pipeline built it, which tests ran, and that the artifact has not been tampered with between build and deployment. In an era where supply chain attacks have become a primary threat vector, this investment directly reduces the blast radius of a compromised build system or malicious dependency.
