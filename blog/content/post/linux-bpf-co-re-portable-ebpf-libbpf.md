BPF CO-RE (Compile Once, Run Everywhere) solved the portability problem that made eBPF programs brittle across kernel versions. Before CO-RE, an eBPF program compiled against one kernel's headers would silently read garbage or crash when run on a kernel where the same struct laid out its fields at different offsets. CO-RE encodes relocation information into the BPF object file and uses BTF (BPF Type Format) to resolve the correct field offsets at load time, matching the compile-time type layout to the runtime kernel's actual layout.

This guide covers the complete CO-RE development workflow: setting up the toolchain, writing BPF C programs that use CO-RE relocations, defining maps with BTF-encoded types, consuming events in userspace with libbpf, and deploying programs across a fleet with heterogeneous kernel versions.

## CO-RE Prerequisites and Toolchain

### Kernel Requirements

CO-RE requires:
- Linux kernel 5.4+ for basic CO-RE support
- Linux kernel 5.8+ for ring buffer maps (`BPF_MAP_TYPE_RINGBUF`)
- BTF enabled in the kernel (`CONFIG_DEBUG_INFO_BTF=y`)
- The running kernel must expose `/sys/kernel/btf/vmlinux`

Verify BTF availability:

```bash
# Check if vmlinux BTF is present
ls -lh /sys/kernel/btf/vmlinux

# Verify BTF contains type information
bpftool btf dump file /sys/kernel/btf/vmlinux format c | head -50

# Check kernel version and BPF feature support
uname -r
bpftool feature probe kernel | grep -E "map_type|prog_type|helper" | head -20
```

### Toolchain Setup

```bash
# Install clang, LLVM, and libbpf development headers
# On Fedora/RHEL:
dnf install clang llvm libbpf-devel bpftool kernel-devel

# On Ubuntu/Debian:
apt-get install clang llvm libbpf-dev linux-tools-$(uname -r)

# Verify clang version (12+ required for full CO-RE support)
clang --version

# Generate vmlinux.h from the running kernel's BTF data
bpftool btf dump file /sys/kernel/btf/vmlinux format c > vmlinux.h
```

The `vmlinux.h` header generated by `bpftool` contains all kernel type definitions as BTF-annotated C structures. This single file replaces the fragmented kernel headers that previously caused portability issues.

## Project Structure

A well-organized libbpf CO-RE project:

```
my-bpf-tool/
├── Makefile
├── vmlinux.h          # Generated: bpftool btf dump ... > vmlinux.h
├── src/
│   ├── execve_monitor.bpf.c    # BPF C program (kernel-side)
│   ├── execve_monitor.h        # Shared structs between kernel and user
│   └── execve_monitor.c        # Userspace program
└── build/
    ├── execve_monitor.bpf.o    # Compiled BPF object
    ├── execve_monitor.skel.h   # Generated skeleton header
    └── execve_monitor          # Final binary
```

### Makefile

```makefile
CLANG ?= clang
CC    ?= gcc
ARCH  := $(shell uname -m | sed 's/x86_64/x86/' | sed 's/aarch64/arm64/')

BPF_CFLAGS := -g -O2 -target bpf -D__TARGET_ARCH_$(ARCH) \
              -I/usr/include/$(shell uname -m)-linux-gnu \
              -Wall -Wno-unused-variable

.PHONY: all clean vmlinux

all: build/execve_monitor

vmlinux:
	bpftool btf dump file /sys/kernel/btf/vmlinux format c > vmlinux.h

build/execve_monitor.bpf.o: src/execve_monitor.bpf.c vmlinux.h
	$(CLANG) $(BPF_CFLAGS) -c $< -o $@

build/execve_monitor.skel.h: build/execve_monitor.bpf.o
	bpftool gen skeleton $< > $@

build/execve_monitor: src/execve_monitor.c build/execve_monitor.skel.h
	$(CC) -g -O2 -Wall -I./build \
	      -o $@ $< -lbpf -lelf -lz

clean:
	rm -rf build/*.o build/*.skel.h build/execve_monitor
```

## Writing the BPF Program (Kernel Side)

### Shared Header

```c
/* src/execve_monitor.h
 * Shared between the BPF program and the userspace consumer.
 */
#pragma once

#define TASK_COMM_LEN 16
#define MAX_FILENAME_LEN 256

struct event {
    __u32 pid;
    __u32 ppid;
    __u32 uid;
    __u32 gid;
    char  comm[TASK_COMM_LEN];
    char  filename[MAX_FILENAME_LEN];
    int   retval;
};
```

### BPF C Program: execve_monitor.bpf.c

```c
/* src/execve_monitor.bpf.c
 *
 * Monitor execve() system calls using BPF CO-RE with libbpf.
 * Demonstrates:
 *   - CO-RE kernel struct access (task_struct)
 *   - Ring buffer map for efficient event delivery
 *   - BTF-annotated map definitions
 */

#include "vmlinux.h"
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_core_read.h>
#include <bpf/bpf_tracing.h>
#include "execve_monitor.h"

/* Ring buffer map — preferred over perf buffer for kernel 5.8+ */
struct {
    __uint(type, BPF_MAP_TYPE_RINGBUF);
    __uint(max_entries, 256 * 1024);
} events SEC(".maps");

/* Per-CPU array to stage event data before committing to ring buffer */
struct {
    __uint(type, BPF_MAP_TYPE_PERCPU_ARRAY);
    __uint(max_entries, 1);
    __type(key, __u32);
    __type(value, struct event);
} staging SEC(".maps");

SEC("tp/syscalls/sys_enter_execve")
int handle_execve_enter(struct trace_event_raw_sys_enter *ctx)
{
    __u32 key = 0;
    struct event *evt;

    evt = bpf_map_lookup_elem(&staging, &key);
    if (!evt)
        return 0;

    __u64 pid_tgid = bpf_get_current_pid_tgid();
    __u64 uid_gid  = bpf_get_current_uid_gid();

    evt->pid = pid_tgid >> 32;
    evt->uid = uid_gid & 0xFFFFFFFF;
    evt->gid = uid_gid >> 32;

    bpf_get_current_comm(&evt->comm, sizeof(evt->comm));

    /* CO-RE read: access task_struct->real_parent->tgid safely across kernels */
    struct task_struct *task = (struct task_struct *)bpf_get_current_task();
    evt->ppid = BPF_CORE_READ(task, real_parent, tgid);

    const char *filename = (const char *)ctx->args[0];
    bpf_probe_read_user_str(evt->filename, sizeof(evt->filename), filename);

    return 0;
}

SEC("tp/syscalls/sys_exit_execve")
int handle_execve_exit(struct trace_event_raw_sys_exit *ctx)
{
    __u32 key = 0;
    struct event *staged;

    staged = bpf_map_lookup_elem(&staging, &key);
    if (!staged)
        return 0;

    __u32 pid = bpf_get_current_pid_tgid() >> 32;
    if (staged->pid != pid)
        return 0;

    staged->retval = ctx->ret;

    struct event *evt = bpf_ringbuf_reserve(&events, sizeof(*evt), 0);
    if (!evt)
        return 0;

    __builtin_memcpy(evt, staged, sizeof(*evt));
    bpf_ringbuf_submit(evt, 0);

    return 0;
}

char LICENSE[] SEC("license") = "GPL";
```

### CO-RE Read Macros Reference

The core macros that make CO-RE work — each inserts a BTF relocation record that libbpf resolves at load time against the running kernel's type information:

```c
/* Direct field read with CO-RE relocation */
__u32 val = BPF_CORE_READ(task, pid);

/* Nested field traversal */
__u32 ppid = BPF_CORE_READ(task, real_parent, tgid);

/* Write into a destination variable */
BPF_CORE_READ_INTO(&dest, task, mm, start_brk);

/* Check if a field exists in the running kernel's BTF */
if (bpf_core_field_exists(task->exit_code)) {
    /* field present in this kernel version */
}

/* String read from kernel space */
bpf_core_read_str(dest, sizeof(dest), src);
```

## Userspace Program with libbpf Skeleton

The `bpftool gen skeleton` command generates a type-safe C header that wraps the BPF object's lifecycle:

```c
/* src/execve_monitor.c */

#include <stdio.h>
#include <signal.h>
#include <unistd.h>
#include <errno.h>
#include <time.h>
#include <bpf/libbpf.h>
#include "execve_monitor.skel.h"
#include "execve_monitor.h"

static volatile int running = 1;

static void sig_handler(int sig)
{
    running = 0;
}

static int handle_event(void *ctx, void *data, size_t data_sz)
{
    const struct event *evt = data;
    struct timespec ts;
    clock_gettime(CLOCK_REALTIME, &ts);

    printf("[%10ld.%06ld] pid=%-6u ppid=%-6u uid=%-4u gid=%-4u "
           "comm=%-16s filename=%s (ret=%d)\n",
           ts.tv_sec, ts.tv_nsec / 1000,
           evt->pid, evt->ppid, evt->uid, evt->gid,
           evt->comm, evt->filename, evt->retval);

    return 0;
}

static int libbpf_print_fn(enum libbpf_print_level level,
                            const char *format, va_list args)
{
    if (level == LIBBPF_DEBUG)
        return 0;
    return vfprintf(stderr, format, args);
}

int main(void)
{
    struct execve_monitor_bpf *skel = NULL;
    struct ring_buffer *rb = NULL;
    int err;

    libbpf_set_print(libbpf_print_fn);
    signal(SIGINT,  sig_handler);
    signal(SIGTERM, sig_handler);

    skel = execve_monitor_bpf__open();
    if (!skel) {
        fprintf(stderr, "Failed to open BPF skeleton: %d\n", errno);
        return 1;
    }

    err = execve_monitor_bpf__load(skel);
    if (err) {
        fprintf(stderr, "Failed to load BPF skeleton: %d\n", err);
        goto cleanup;
    }

    err = execve_monitor_bpf__attach(skel);
    if (err) {
        fprintf(stderr, "Failed to attach BPF programs: %d\n", err);
        goto cleanup;
    }

    rb = ring_buffer__new(bpf_map__fd(skel->maps.events),
                          handle_event, NULL, NULL);
    if (!rb) {
        err = -errno;
        fprintf(stderr, "Failed to create ring buffer: %d\n", err);
        goto cleanup;
    }

    printf("Monitoring execve() syscalls. Press Ctrl-C to stop.\n");

    while (running) {
        err = ring_buffer__poll(rb, 100);
        if (err == -EINTR) {
            err = 0;
            break;
        }
        if (err < 0) {
            fprintf(stderr, "Ring buffer polling error: %d\n", err);
            break;
        }
    }

cleanup:
    ring_buffer__free(rb);
    execve_monitor_bpf__destroy(skel);
    return err < 0 ? -err : 0;
}
```

## Advanced CO-RE Patterns

### Handling Kernel Version Differences

Some kernel structs reorganize fields between versions. CO-RE handles this transparently:

```c
SEC("kprobe/tcp_v4_connect")
int BPF_KPROBE(trace_tcp_connect, struct sock *sk)
{
    __u32 zero = 0;
    struct event *evt = bpf_map_lookup_elem(&staging, &zero);
    if (!evt)
        return 0;

    /* These reads work across kernel versions where sk_common fields
     * may be at different positions. CO-RE resolves the correct offsets. */
    __u32 saddr = BPF_CORE_READ(sk, __sk_common.skc_rcv_saddr);
    __u32 daddr = BPF_CORE_READ(sk, __sk_common.skc_daddr);
    __u16 dport = BPF_CORE_READ(sk, __sk_common.skc_dport);
    __u16 sport = BPF_CORE_READ(sk, __sk_common.skc_num);

    evt->pid   = bpf_get_current_pid_tgid() >> 32;
    evt->dport = bpf_ntohs(dport);
    evt->sport = sport;

    return 0;
}
```

### BTF Maps for Rich Type Information

BTF-annotated maps allow `bpftool` and other tools to inspect map keys and values with full type information:

```c
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 10240);
    __type(key,   __u32);
    __type(value, struct event);
} active_pids SEC(".maps");

struct {
    __uint(type, BPF_MAP_TYPE_PERCPU_HASH);
    __uint(max_entries, 1024);
    __type(key,   struct flow_key);
    __type(value, __u64);
} packet_counts SEC(".maps");
```

Inspect BTF-annotated maps at runtime:

```bash
bpftool map list
bpftool map dump name active_pids
bpftool map show name active_pids --pretty
```

```yaml

kind: DaemonSet
metadata:
  name: execve-monitor
  namespace: security
spec:
  selector:
    matchLabels:
      app: execve-monitor
  template:
    metadata:
      labels:
        app: execve-monitor
    spec:
      hostPID: true
      hostNetwork: true
      tolerations:
      - effect: NoSchedule
        operator: Exists
      - effect: NoExecute
        operator: Exists
      containers:
      - name: monitor
        image: registry.example.com/execve-monitor:v1.3.0
        securityContext:
          privileged: true
          capabilities:
            add:
            - SYS_ADMIN
            - BPF
            - PERFMON
        volumeMounts:
        - name: sys-kernel-btf
          mountPath: /sys/kernel/btf
          readOnly: true
        - name: sys-kernel-debug
          mountPath: /sys/kernel/debug
          readOnly: true

          mountPath: /proc
          readOnly: true
        resources:
          requests:
            cpu: "100m"
            memory: "64Mi"
          limits:
            cpu: "500m"
            memory: "256Mi"

      - name: sys-kernel-btf
```

## Debugging CO-RE Programs

### Verifier Errors

```bash
# Enable verbose verifier output during development
sudo bpftool prog load execve_monitor.bpf.o /sys/fs/bpf/execve_monitor \
  type tracepoint 2>&1 | head -100

# Check BTF relocation log
sudo bpftool prog load execve_monitor.bpf.o /sys/fs/bpf/test \
  -d 2>&1 | grep -E "CO-RE|reloc|field"

# Verify the struct field exists in the running kernel's BTF
bpftool btf dump file /sys/kernel/btf/vmlinux | \
  grep -A 5 "task_struct" | head -30
```

### Ring Buffer Sizing

Monitor ring buffer drop rates to detect backpressure:

```bash

# Check for dropped events in tracing infrastructure
cat /sys/kernel/debug/tracing/per_cpu/cpu0/stats
```

## Performance Characteristics

CO-RE overhead compared to alternatives:

| Approach | Overhead | Portability | Safety |
|----------|----------|-------------|--------|
| BPF CO-RE (libbpf) | ~100-200ns/event | Kernel 5.4+ | Verifier enforced |
| BPF without CO-RE | ~100-200ns/event | Single kernel | Verifier enforced |
| ftrace/tracefs | ~500ns/event | All kernels | Limited |
| ptrace | ~5-10 us/event | All kernels | None |
| kernel module | ~50-100ns/event | ABI coupling | Ring 0 |

CO-RE programs on production nodes typically consume 0.1-0.5% of a single CPU core for moderate event rates (10,000 events/second). Ring buffer consumers add negligible overhead compared to the BPF program's probe execution cost.

The libbpf skeleton pattern (`bpftool gen skeleton`) provides compile-time type safety between the BPF object's maps and programs and the userspace consumer, eliminating an entire class of runtime type mismatch errors that plagued the older `bpf()` syscall wrapping approach.

### CO-RE in Production: Operational Considerations

**Version skew between nodes**: In clusters with mixed kernel versions, test CO-RE programs on the oldest kernel in the fleet. BTF-absent kernels (pre-5.4, or kernels built without `CONFIG_DEBUG_INFO_BTF`) require embedding a pre-built BTF blob using `LIBBPF_OPTS(bpf_object_open_opts, opts, .btf_custom_path = ...)`.

**Container image size**: The compiled BPF object and skeleton are typically under 500KB. The libbpf shared library adds ~500KB. Full monitoring agent images including the userspace binary are generally under 20MB.

**Graceful shutdown**: CO-RE programs loaded via the skeleton API are automatically unloaded when `execve_monitor_bpf__destroy(skel)` is called, or when the process exits. The kernel reference-counts BPF program attachments, so a crashed process leaves no orphaned probes.
