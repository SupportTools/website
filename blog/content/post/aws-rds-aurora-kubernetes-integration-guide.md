Running a Kubernetes application against AWS RDS Aurora requires solving several non-trivial problems: Kubernetes pods scale horizontally and each pod opens database connections, which quickly exhausts PostgreSQL's connection limit. Rotating database credentials safely in a running cluster is operationally painful. And provisioning RDS instances via ClickOps or Terraform outside the cluster creates a gap in your GitOps workflow.

This guide addresses all three problems. It covers RDS Proxy for connection pooling to handle connection spikes from Kubernetes, IAM database authentication to eliminate static passwords, the ACK RDS Controller to provision Aurora clusters as Kubernetes resources, and production monitoring with Enhanced Monitoring and CloudWatch.

# AWS RDS Aurora and Kubernetes: Connection Pooling, IAM Auth, and Operator Patterns

## The Connection Pooling Problem in Kubernetes

PostgreSQL has a fixed `max_connections` parameter. Each connection consumes memory and file descriptors on the database server. Aurora PostgreSQL defaults to `max_connections` based on instance size:

```
db.r7g.large (32GB): ~450 max_connections
db.r7g.xlarge (64GB): ~900 max_connections
db.r7g.4xlarge (256GB): ~3600 max_connections
```

In Kubernetes, if you have 50 pods each opening a connection pool of 10 connections, you hit 500 connections — exhausting a `db.r7g.large` before considering other workloads, migrations, or administrative connections.

The solution is connection pooling at the application layer (PgBouncer) or at the AWS-managed layer (RDS Proxy).

## RDS Proxy: Architecture and Setup

RDS Proxy sits between your application and the RDS cluster. It maintains a pool of long-lived connections to the database and multiplexes thousands of application connections onto that pool:

```
Kubernetes Pods (500 connections)
    │
    ▼
RDS Proxy (endpoint: proxy.xxxxxx.us-east-1.rds.amazonaws.com)
    │  (maintained pool of ~50 connections)
    ▼
Aurora PostgreSQL Writer Endpoint
```

```bash
# Create an RDS Proxy via AWS CLI
aws rds create-db-proxy \
  --db-proxy-name production-aurora-proxy \
  --engine-family POSTGRESQL \
  --auth '[
    {
      "AuthScheme": "SECRETS",
      "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789012:secret/prod/aurora-credentials",
      "IAMAuth": "REQUIRED"
    }
  ]' \
  --role-arn arn:aws:iam::123456789012:role/rds-proxy-role \
  --vpc-subnet-ids subnet-12345678 subnet-87654321 subnet-11223344 \
  --vpc-security-group-ids sg-12345678 \
  --require-tls \
  --idle-client-timeout 1800 \
  --debug-logging false

# Register the proxy with the Aurora cluster
aws rds register-db-proxy-targets \
  --db-proxy-name production-aurora-proxy \
  --db-cluster-identifiers aurora-production-cluster

# Get the proxy endpoint
aws rds describe-db-proxies \
  --db-proxy-name production-aurora-proxy \
  --query 'DBProxies[0].Endpoint' \
  --output text
# production-aurora-proxy.proxy-xxxxxxxxxxxxxx.us-east-1.rds.amazonaws.com
```

## IAM Database Authentication in Go

IAM database authentication replaces the database password with a short-lived token generated by the AWS SDK. The token expires after 15 minutes, so your application generates a fresh token for each new connection:

```go
// pkg/db/iam_auth.go
package db

import (
	"context"
	"database/sql"
	"database/sql/driver"
	"fmt"
	"net"
	"net/url"
	"time"

	"github.com/aws/aws-sdk-go-v2/aws"
	"github.com/aws/aws-sdk-go-v2/config"
	"github.com/aws/aws-sdk-go-v2/feature/rds/auth"
	"github.com/lib/pq"
)

// IAMAuthConfig holds connection parameters for IAM-authenticated RDS/Aurora.
type IAMAuthConfig struct {
	Host     string // RDS Proxy endpoint or Aurora endpoint
	Port     int
	DBName   string
	User     string // Database user that has been granted rds_iam role
	Region   string
	MaxConns int
	MinConns int
}

// iamConnector implements driver.Connector. It generates a fresh IAM token
// for each new connection, which is required because tokens expire after 15 minutes.
type iamConnector struct {
	cfg       IAMAuthConfig
	awsConfig aws.Config
	driver    driver.Driver
}

func (c *iamConnector) Connect(ctx context.Context) (driver.Conn, error) {
	token, err := auth.BuildAuthToken(
		ctx,
		fmt.Sprintf("%s:%d", c.cfg.Host, c.cfg.Port),
		c.cfg.Region,
		c.cfg.User,
		c.awsConfig.Credentials,
	)
	if err != nil {
		return nil, fmt.Errorf("building IAM auth token: %w", err)
	}

	// Build the PostgreSQL DSN with the IAM token as the password.
	dsn := url.URL{
		Scheme: "postgres",
		User:   url.UserPassword(c.cfg.User, token),
		Host:   net.JoinHostPort(c.cfg.Host, fmt.Sprintf("%d", c.cfg.Port)),
		Path:   "/" + c.cfg.DBName,
	}

	// sslmode=require is mandatory for IAM auth — the token is only secure over TLS.
	q := dsn.Query()
	q.Set("sslmode", "require")
	q.Set("connect_timeout", "10")
	dsn.RawQuery = q.Encode()

	return c.driver.(driver.DriverContext).OpenConnector(dsn.String()).(interface {
		Connect(context.Context) (driver.Conn, error)
	}).Connect(ctx)
}

func (c *iamConnector) Driver() driver.Driver {
	return c.driver
}

// NewIAMAuthDB creates a *sql.DB that generates fresh IAM tokens for each connection.
func NewIAMAuthDB(ctx context.Context, cfg IAMAuthConfig) (*sql.DB, error) {
	awsCfg, err := config.LoadDefaultConfig(ctx,
		config.WithRegion(cfg.Region),
	)
	if err != nil {
		return nil, fmt.Errorf("loading AWS config: %w", err)
	}

	connector := &iamConnector{
		cfg:       cfg,
		awsConfig: awsCfg,
		driver:    &pq.Driver{},
	}

	db := sql.OpenDB(connector)
	db.SetMaxOpenConns(cfg.MaxConns)
	db.SetMaxIdleConns(cfg.MinConns)
	// Set connection max lifetime shorter than the 15-minute token expiry.
	// This forces the connector to generate fresh tokens for new connections.
	db.SetConnMaxLifetime(10 * time.Minute)
	db.SetConnMaxIdleTime(5 * time.Minute)

	if err := db.PingContext(ctx); err != nil {
		return nil, fmt.Errorf("pinging database: %w", err)
	}

	return db, nil
}
```


## Aurora Serverless v2 Configuration

Aurora Serverless v2 scales Aurora capacity units (ACUs) between a minimum and maximum, eliminating the need to choose an instance type. Each ACU is approximately 2GB of memory with corresponding CPU:

```bash
# Create Aurora Serverless v2 cluster
aws rds create-db-cluster \
  --db-cluster-identifier aurora-serverless-v2-prod \
  --engine aurora-postgresql \
  --engine-version 16.2 \
  --master-username postgres \
  --manage-master-user-password \
  --serverless-v2-scaling-configuration MinCapacity=0.5,MaxCapacity=32 \
  --vpc-security-group-ids sg-12345678 \
  --db-subnet-group-name production-db-subnet-group \
  --backup-retention-period 7 \
  --deletion-protection \
  --enable-cloudwatch-logs-exports '["postgresql","upgrade"]' \
  --storage-encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/xxxxxxxx

# Add a Serverless v2 reader instance
aws rds create-db-instance \
  --db-instance-identifier aurora-serverless-v2-prod-reader-1 \
  --db-cluster-identifier aurora-serverless-v2-prod \
  --db-instance-class db.serverless \
  --engine aurora-postgresql

# Monitor current ACU usage
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name ServerlessDatabaseCapacity \
  --dimensions Name=DBClusterIdentifier,Value=aurora-serverless-v2-prod \
  --start-time $(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 60 \
  --statistics Average \
  --query 'Datapoints[*].[Timestamp,Average]' \
  --output table
```

### Read Replica Routing in Go

Use separate connection pools for reads and writes to leverage Aurora's reader endpoint:

```go
// pkg/db/aurora.go
package db

import (
	"context"
	"database/sql"
	"fmt"
)

// AuroraDB provides separate read and write connection pools,
// routing queries to the appropriate Aurora endpoint.
type AuroraDB struct {
	writer *sql.DB
	reader *sql.DB
}

// NewAuroraDB creates connection pools for both writer and reader endpoints.
func NewAuroraDB(ctx context.Context, region string) (*AuroraDB, error) {
	// Writer endpoint — only the current primary instance
	writer, err := NewIAMAuthDB(ctx, IAMAuthConfig{
		Host:     "aurora-prod-cluster.cluster-xxxxxxxx.us-east-1.rds.amazonaws.com",
		Port:     5432,
		DBName:   "production",
		User:     "app_user",
		Region:   region,
		MaxConns: 20,
		MinConns: 5,
	})
	if err != nil {
		return nil, fmt.Errorf("writer connection: %w", err)
	}

	// Reader endpoint — load-balanced across all reader instances
	reader, err := NewIAMAuthDB(ctx, IAMAuthConfig{
		Host:     "aurora-prod-cluster.cluster-ro-xxxxxxxx.us-east-1.rds.amazonaws.com",
		Port:     5432,
		DBName:   "production",
		User:     "app_user",
		Region:   region,
		MaxConns: 40,  // Readers can handle more connections
		MinConns: 10,
	})
	if err != nil {
		writer.Close()
		return nil, fmt.Errorf("reader connection: %w", err)
	}

	return &AuroraDB{writer: writer, reader: reader}, nil
}

// Writer returns the connection pool for write operations.
// Use for INSERT, UPDATE, DELETE, and reads that must see the latest data.
func (a *AuroraDB) Writer() *sql.DB { return a.writer }

// Reader returns the connection pool for read-only operations.
// Queries here may lag behind the writer by a few milliseconds (replica lag).
func (a *AuroraDB) Reader() *sql.DB { return a.reader }

func (a *AuroraDB) Close() {
	a.writer.Close()
	a.reader.Close()
}

// ExecWrite runs a query on the writer endpoint.
func (a *AuroraDB) ExecWrite(ctx context.Context, query string, args ...interface{}) (sql.Result, error) {
	return a.writer.ExecContext(ctx, query, args...)
}

// QueryRead runs a query on the reader endpoint.
func (a *AuroraDB) QueryRead(ctx context.Context, query string, args ...interface{}) (*sql.Rows, error) {
	return a.reader.QueryContext(ctx, query, args...)
}
```

## ACK RDS Controller: Operator-Driven Provisioning

The AWS Controllers for Kubernetes (ACK) provide Kubernetes operators for AWS services. The RDS controller lets you provision Aurora clusters as Kubernetes Custom Resources, keeping your database infrastructure in Git alongside your application manifests:

```bash
# Install the ACK RDS controller
helm install ack-rds-controller \
  oci://public.ecr.aws/aws-controllers-k8s/rds-chart \
  --version 1.4.2 \
  --namespace ack-system \
  --create-namespace \
  --set aws.region=us-east-1 \
  --set serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=arn:aws:iam::123456789012:role/ack-rds-controller
```

```yaml
# aurora-cluster.yaml — Aurora cluster as a Kubernetes resource
apiVersion: rds.services.k8s.aws/v1alpha1
kind: DBCluster
metadata:
  name: aurora-production
  namespace: databases
spec:
  dbClusterIdentifier: aurora-production
  engine: aurora-postgresql
  engineVersion: "16.2"

  # Credentials — references an existing Secrets Manager secret
  masterUsername: postgres
  manageMasterUserPasswordRef:
    name: aurora-production-master-secret

  # Serverless v2 scaling
  serverlessV2ScalingConfiguration:
    minCapacity: 0.5
    maxCapacity: 32

  # Network configuration
  dbSubnetGroupName: production-db-subnet-group
  vpcSecurityGroupIDs:
    - sg-12345678

  # Backup and maintenance
  backupRetentionPeriod: 7
  preferredBackupWindow: "02:00-03:00"
  preferredMaintenanceWindow: "sun:04:00-sun:05:00"

  # CloudWatch Logs

    - postgresql
```

## Secrets Management for Database Credentials

Never put database passwords in Kubernetes Secrets directly — they are base64-encoded and visible in etcd. Use External Secrets Operator to sync from AWS Secrets Manager:

```yaml
# external-secret.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: aurora-credentials
  namespace: production
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: aurora-credentials
    creationPolicy: Owner
    template:
      type: Opaque
      engineVersion: v2
      data:
        # Parse the Secrets Manager JSON and extract individual fields
        POSTGRES_HOST: "{{ .proxy_endpoint }}"
        POSTGRES_PORT: "5432"
        POSTGRES_DB: "{{ .dbname }}"
        POSTGRES_USER: "{{ .username }}"
        # For IAM auth, no password is needed — token is generated at runtime
  data:
    - secretKey: proxy_endpoint
      remoteRef:
        key: prod/aurora-credentials
        property: proxy_endpoint
    - secretKey: dbname
      remoteRef:
        key: prod/aurora-credentials
        property: dbname
    - secretKey: username
      remoteRef:
        key: prod/aurora-credentials
        property: username
```

## IAM Auth Token Refresh: Application-Level Health Check

```go
// pkg/db/healthcheck.go
package db

import (
	"context"
	"database/sql"
	"fmt"
	"time"
)

// DBHealthChecker provides periodic health checks and metrics for the DB pool.
type DBHealthChecker struct {
	writer *sql.DB
	reader *sql.DB
}

// Stats returns current connection pool statistics.
func (h *DBHealthChecker) Stats() map[string]interface{} {
	ws := h.writer.Stats()
	rs := h.reader.Stats()
	return map[string]interface{}{
		"writer": map[string]int{
			"open_connections":  ws.OpenConnections,
			"in_use":            ws.InUse,
			"idle":              ws.Idle,
			"wait_count":        int(ws.WaitCount),
			"max_open_conns":    ws.MaxOpenConnections,
		},
		"reader": map[string]int{
			"open_connections": rs.OpenConnections,
			"in_use":           rs.InUse,
			"idle":             rs.Idle,
			"wait_count":       int(rs.WaitCount),
		},
	}
}

// Ping checks connectivity to both endpoints.
func (h *DBHealthChecker) Ping(ctx context.Context) error {
	pingCtx, cancel := context.WithTimeout(ctx, 3*time.Second)
	defer cancel()

	if err := h.writer.PingContext(pingCtx); err != nil {
		return fmt.Errorf("writer ping: %w", err)
	}

	if err := h.reader.PingContext(pingCtx); err != nil {
		return fmt.Errorf("reader ping: %w", err)
	}

	return nil
}
```

## Summary

Integrating Aurora with Kubernetes at production scale requires solving four distinct problems:

1. **Connection pooling**: Use RDS Proxy to multiplex thousands of pod connections onto a smaller pool, preventing connection exhaustion. Configure `MaxConns` per pod at 10-20, not the PostgreSQL default of 100.

2. **Authentication**: Use IAM database authentication + IRSA to eliminate static passwords. The `iamConnector` pattern generates fresh 15-minute tokens per connection, and `ConnMaxLifetime` of 10 minutes ensures no connection ever uses an expired token.

3. **Infrastructure as Code**: Use the ACK RDS Controller to declare your Aurora clusters as Kubernetes CustomResources. This brings database provisioning into your GitOps workflow with the same review and approval process as application deployments.

4. **Observability**: CloudWatch Enhanced Monitoring gives per-second granularity for OS metrics. CloudWatch alarms catch connection exhaustion and replica lag before they impact users. The CloudWatch exporter bridges Aurora metrics into your Prometheus/Grafana stack for unified dashboards.

Aurora Serverless v2 simplifies the instance sizing problem entirely — set a minimum ACU for baseline cost and a maximum ACU for your peak capacity ceiling, and Aurora handles the rest.
